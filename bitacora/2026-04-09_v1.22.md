## PASO 2 — Sincronización de XP y Resumen (Bug V2 / V13)

### Problema
`Summary.tsx` mostraba factores hardcodeados incorrectos:
- `km × 10 XP` → debería ser Runner=12, Roller=8, Ciclista=5
- `min × 2 XP/min` → el factor real es 1 XP/min
- `totalWithBonus = xp + missionBonusXp` → **double-count**: `xp` ya incluía el bonus multiplicado

### Raíz del bug (double-count)
`calcTotalXp` calcula: `xp = round((km×factor + min + missionBonus + encounterBonus) × mult)`  
`Summary` hacía: `total = xp + missionBonusXp` → sumaba el bonus una segunda vez sin multiplicar.

### Archivos modificados
- `src/utils/xpLogic.ts`
- `src/Summary.tsx`

### Cambios realizados

**xpLogic.ts**
- Añadida constante `TIME_XP_PER_MIN = 1` (exportada)
- `calcTotalXp` ahora usa `minutes * TIME_XP_PER_MIN` explícitamente

**Summary.tsx**
- Importa `CLASS_XP_FACTORS` y `TIME_XP_PER_MIN` desde `xpLogic.ts`
- Calcula `factor = CLASS_XP_FACTORS[userClass]` (Runner=12, Roller=8, Ciclista=5)
- Muestra `km × {factor} XP/km` (correcto por tribu)
- Muestra `min × {TIME_XP_PER_MIN} XP/min` (correcto, desde constante)
- Bonus de misiones aparece como línea visual separada (naranja, con border)
- `TOTAL = xp` siempre — el valor autoritativo de `calcTotalXp`, sin doble suma
- Fórmula visual: componentes pre-multiplicador → separador `× Vuelta (×mult)` → TOTAL

### Ejemplo post-fix (Runner, 2 km, 15 min, ×1.2, sin misiones)
```
2.00 km × 12 XP/km    → +24 XP
15m × 1 XP/min        → +15 XP
Vuelta Runner (×1.2)  → ×1.2
TOTAL                 → 47 XP   [round((24+15)×1.2)]
```

## PASO 5 — Debug: 0 misiones visibles pese a 153 en Supabase

### Diagnóstico
Tres bugs independientes en `useMissions.ts` y `types/missions.ts`:

**Bug A — `eq` es case-sensitive en PostgREST**
```
Código:   tribe.eq.Ciclista    ← no matchea 'ciclista' en DB
Fix:      tribe.ilike.ciclista  ← case-insensitive
```

**Bug B — `'all'` vs `'Todas'`: 30 misiones invisibles**
```
Código:   tribe.eq.Todas       ← no matchea tribe='all' en DB
Fix:      tribe.ilike.all,tribe.ilike.todas
```

**Bug C — `normalizeDifficulty` pasaba valores crudos de DB**
```typescript
// ANTES:
if (m.difficulty) return m.difficulty; // devuelve 'épica' si eso tiene la DB

// DESPUÉS: normaliza raw.toLowerCase() antes de switchear
'épica' | 'epica' | 'hard' | 'dificil' → siempre devuelve 'hard'
```

**Bug D — filtro de tipo sin normalizar casing**
```typescript
// ANTES: t !== filters.type  → 'Distance' !== 'distance' = excluido
// DESPUÉS: t.toLowerCase().trim() !== filters.type
```

### Archivos modificados
- `src/hooks/useMissions.ts` — añadido helper `tribeOrFilter()`, aplicado a los 3 queries del archivo
- `src/types/missions.ts` — `normalizeDifficulty` reescrita para manejar todos los casos

### Archivos modificados
- `src/hooks/useMissions.ts`
- `src/components/MissionsScreen.tsx`

### Cambios en `useMissions.ts`
- Query: sin `.or(tribeOrFilter(...))` — solo `SELECT * ORDER BY created_at DESC`
- Guard: solo requiere `userId` (no `tribe`)
- `filteredMissions = missions` — sin filtrado, pasa todo
- Dependency array del `useCallback`: `[userId]` (antes `[tribe, userId]`)
- Debug: `console.log('Misiones cargadas:', mData?.length)`

### Cambios en `MissionsScreen.tsx`
- Chips de dificultad y tipo: **solo resaltadores visuales, no filtran**

  - Cards que no coinciden: `opacity: 0.45` pero **visibles y scrolleables**
- Eliminado `setFilters` del hook (ya no aplica)
- Renderizado: `missions.map(...)` directo — todas sin excepción
- Mensaje vacío cambiado: "Sin misiones en la base de datos" (no "con estos filtros")

**3 badges nuevos por card:**

  ↑          ↑          ↑
